# Future Development: RAG-Based Context Optimization for AI Scan Modules

## Problem

Diana's AI agent scanners (sqli_agent, xss_agent, discovery_agent) use LangGraph ReAct agents with multi-turn conversations. Each turn sends the **full conversation history** back to the LLM — system prompt, scope constraints, all prior HTTP requests/responses, all prior tool calls and results.

This means:
- **Token cost grows quadratically** — turn N re-sends everything from turns 1 through N-1
- **Context gets diluted** — by turn 15, the LLM is processing thousands of tokens of old HTTP responses that are no longer relevant to the current probe
- **Redundant context across modules** — the sqli_agent and xss_agent both receive the full sitemap, tech stack, and prior findings, even when only a small subset is relevant to the current endpoint
- **Identical requests across scans** — rescanning the same app rebuilds all context from scratch, even if 90% of the endpoints haven't changed

## Proposed Solutions

### 1. Response Caching (Low-Hanging Fruit)

**What:** Cache HTTP responses by (URL, method, auth_level, body_hash) during a scan. If an agent requests an endpoint that was already fetched (by any module), return the cached response without making a new HTTP request.

**Where it helps:**
- Discovery agent probes `/api/users` → sqli_agent later probes the same endpoint → cache hit, no HTTP call
- The `_seen_requests` set in `tool_agent.py` already deduplicates within a single agent run, but not across modules

**Implementation:**
- Redis (already in the stack) as response cache with TTL matching scan duration
- Key: `scan:{scan_id}:response:{hash(method|url|auth|body)}`
- Value: `{status_code, headers, body_preview}`
- The `http_request` tool checks cache before making a live request

**Token savings:** Modest — reduces HTTP calls but doesn't reduce LLM context. Main value is reducing load on the target.

### 2. Sliding Window Context (Medium Effort)

**What:** Instead of sending the full conversation history to the LLM on each turn, maintain a sliding window of the last N turns plus a **summary of earlier turns**.

**How it works:**
```
Turn 1-5:   Full history sent (small enough)
Turn 6:     Turns 1-3 summarized into 2-3 sentences, turns 4-6 sent in full
Turn 10:    Turns 1-7 summarized, turns 8-10 sent in full
Turn 15:    Turns 1-12 summarized, turns 13-15 sent in full
```

The summary is generated by the LLM itself (one cheap call) or by a deterministic extractor that pulls out:
- Endpoints tested and their responses (pass/fail/interesting)
- Payloads that were blocked vs. payloads that got through
- Findings reported so far

**Token savings:** Significant — prevents the quadratic blowup. A 25-turn XSS agent run currently sends ~25x the system prompt alone. With sliding window, the context stays roughly constant after turn 5.

**Risk:** The LLM might lose track of what it already tried. Mitigated by keeping the "already tested" list in the summary.

### 3. RAG for Scan Context (Higher Effort, Highest Payoff)

**What:** Replace the monolithic conversation context with a vector store that the agent queries on each turn. The agent retrieves only the context relevant to its current decision.

**Architecture:**
```
┌─────────────────────────────────────────────────┐
│  Vector Store (per-scan, ephemeral)              │
│                                                  │
│  Collections:                                    │
│  ├── endpoints    — URL, method, params, tech    │
│  ├── responses    — HTTP responses by endpoint   │
│  ├── findings     — confirmed vulns so far       │
│  ├── payloads     — what was tried, what worked  │
│  └── app_context  — tech stack, auth flow, WAF   │
└──────────────┬──────────────────────────────────┘
               │ retrieve relevant context
               ▼
┌──────────────────────────────────────────────────┐
│  Agent Turn N                                     │
│                                                   │
│  System prompt (static, small)                    │
│  + Retrieved context:                             │
│    - This endpoint's prior responses (2-3)        │
│    - Similar endpoints' findings (if any)         │
│    - Payloads that worked on similar params       │
│    - App tech stack (once, cached)                │
│  + Current task: "Test /api/users?id= for SQLi"   │
└──────────────────────────────────────────────────┘
```

**What goes into the vector store:**
- **After crawling:** All endpoints, forms, parameters, tech stack observations
- **After each HTTP request:** The response (status, headers, body preview), indexed by endpoint + parameter
- **After each finding:** The finding details, indexed by vuln type + endpoint pattern
- **After each payload attempt:** What was tried, what was blocked, what got through

**What the agent retrieves per turn:**
- "Give me all prior responses for this endpoint" (exact match)
- "Give me findings from similar endpoints" (semantic search — e.g., other `/api/*` endpoints with ID parameters)
- "Give me payloads that bypassed filters on this app" (semantic search)

**Token savings:** Dramatic. Instead of sending 20 prior turns (maybe 8,000 tokens), the agent gets ~1,000 tokens of highly relevant context. And it improves with scale — the more endpoints tested, the better the retrieval gets.

**Cross-module intelligence:** The sqli_agent's findings are immediately available to the xss_agent via the shared vector store. Currently this happens via the DB (`store_finding` / `get_findings_summary`), but RAG makes it semantic — "the sqli_agent found that `/api/users` reflects input unescaped" is directly useful context for the xss_agent testing the same endpoint.

### 4. Cross-Scan Caching (Longest Horizon)

**What:** Persist the vector store across scans of the same target. On rescan, the agent starts with knowledge of what worked last time.

**What this enables:**
- "Last scan found SQLi on `/api/orders?id=` — verify it's still present" (regression check, 1 turn instead of 10)
- "Last scan's XSS payloads were all blocked by CSP — try CSP bypass techniques first" (learned strategy)
- "These 40 endpoints were clean last scan and haven't changed — skip them" (delta scanning)

**Risk:** Stale context. The app may have changed. Mitigate by:
- TTL on cached context (expire after N days or N scans)
- Always verify cached findings with a fresh probe
- Store the app's content hash to detect changes

## Implementation Path

| Phase | What | Token Reduction | Effort |
|-------|------|----------------|--------|
| A | Response caching in Redis | ~10% (fewer HTTP calls, same LLM context) | Low — modify `http_request` tool |
| B | Sliding window context | ~40-60% (prevents quadratic growth) | Medium — modify `ToolUsingAgent.run()` |
| C | RAG vector store per scan | ~70-80% (only relevant context per turn) | High — new component, LangChain retriever integration |
| D | Cross-scan persistence | ~30% on rescans (skip known-clean endpoints) | Medium — persist and load vector store |

## Technology Choices

**Vector store options (all work with LangChain):**
- **ChromaDB** — in-process, ephemeral by default, perfect for per-scan stores. No infrastructure needed.
- **pgvector** — we already have PostgreSQL. Add the extension, store embeddings alongside scan data. Good for cross-scan persistence.
- **FAISS** — fast, in-memory, good for per-scan use. No persistence without explicit save/load.

**Embedding model:**
- Use Bedrock's Titan Embeddings (already have Bedrock access) or a local model via Ollama
- Embeddings are cheap — the whole point is to replace expensive LLM context with cheap embedding lookups

**Recommended:** Start with ChromaDB for per-scan RAG (Phase C), migrate to pgvector for cross-scan persistence (Phase D).

## Integration Points

The changes are localized to a few files:

| File | Change |
|------|--------|
| `src/diana/ai/tool_agent.py` | Add retriever to agent loop, replace full history with retrieved context |

| `src/diana/scanners/*_agent.py` | Index findings and responses into vector store |

## Relationship to Agent Team

This optimization is **orthogonal to the agent team** — it improves individual scan efficiency, which benefits every agent that runs scans (Validation, Benchmark). The cost savings compound:

- Each iteration runs 2-3 scans
- Each scan has 3 AI agent modules with 20-25 turns each
- RAG reduces per-turn tokens by ~70%
- Net effect: **iteration cost drops by ~50-60%**

The Benchmark Agent would track token usage before/after to validate the savings.

## Open Questions

1. **Embedding quality for security payloads** — will generic embeddings capture the semantic similarity between `' OR 1=1--` and `" OR "a"="a`? May need domain-specific embeddings or keyword-based retrieval as fallback.
2. **Latency tradeoff** — each turn now includes an embedding lookup. Is the latency hit worth the token savings? (Probably yes — LLM calls are 1-5s, embedding lookups are <100ms.)
3. **Context relevance** — how do we measure whether the retrieved context is actually useful? Could track "did the agent use the retrieved context" via tool call analysis.
